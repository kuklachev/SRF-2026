# Аналитика проекта «Вертикаль» — мобильное приложение для записи на тренировки в скалодроме

Учебный (демонстрационный) проект по бизнес- и системному анализу, показывающий полный путь от первоначального брифа заказчика до готового комплекта технической документации и API-контракта для клиентского мобильного приложения скалодрома.

> **Важно.** Все материалы, представленные в этом репозитории, были подготовлены с использованием бесплатных веб-версий **ChatGPT** и **Claude** — без платных подписок, API и специализированных no-code или аналитических сервисов. Все промпты, применявшиеся при создании документов, собраны в файле [`prompts/my-prompts.md`](./prompts/my-prompts.md).

## Описание проекта

Владелица скалодрома «Вертикаль» сейчас ведёт запись клиентов на групповые тренировки вручную — через Telegram и бумажный журнал. Для автоматизации процесса ей требуется мобильное приложение, которое позволит посетителям самостоятельно записываться на занятия.

При этом существующая инфраструктура скалодрома (серверная часть, расписание и административная панель) уже реализована. Поэтому объектом анализа и проектирования является **исключительно клиентское мобильное приложение и API, предназначенное для его работы**.

Материалы проекта демонстрируют полный цикл аналитической работы:

1. выявление требований на основе первоначального брифа заказчика;
2. формализацию бизнес-, функциональных и нефункциональных требований;
3. подготовку постановок для дизайна экранов;
4. проектирование модели данных и взаимодействия через API;
5. формирование итогового комплекта технических заданий для разработки мобильного приложения;
6. разработку спецификации API в формате OpenAPI.

## Структура репозитория

```text
analysis/
├── 0-customer-brief/       # исходный (намеренно неполный) бриф заказчика
├── 1-elicitation/          # выявление требований: вопросы заказчику + описание домена
├── 2-requirements/         # формализованные требования
├── 3-design-brief/         # постановки на дизайн экранов
├── 4-design/               # модель данных и sequence-диаграммы API
├── 5-mobile-app-spec/      # итоговый пакет ТЗ для мобильного приложения
├── api/                    # контракт API (OpenAPI)
└── prompts/                # промпты, которыми получены все материалы
```

### `0-customer-brief/`

* **brief-climbing.md** — первоначальный бриф от заказчика (Оли, владелицы скалодрома), содержащий описание существующей проблемы. Документ специально оставлен неполным и местами противоречивым, чтобы использовать его как отправную точку аналитической работы.

### `1-elicitation/`

* **customer-questions.md** — перечень уточняющих вопросов заказчику и полученные в ходе обсуждения ответы.
* **domain-description.md** — структурированное описание предметной области, сформированное на основе исходного брифа и результатов интервьюирования заказчика.

### `2-requirements/`

* **business-requirements.md** — бизнес-требования (BR) с уникальными идентификаторами, приоритетами и ссылками на источники.
* **functional-requirements.md** — функциональные требования (FR), описывающие возможности системы.
* **non-functional-requirements.md** — нефункциональные требования (NFR), определяющие измеримые показатели качества системы, а не общие формулировки.
* **use-cases.md** — сценарии использования (UC).
* **user-stories.md** — пользовательские истории (US).

### `3-design-brief/`

* **00-screens-registry.md** — полный реестр экранов мобильного приложения.
* **01…09-*.md** — постановки для дизайна каждого экрана: список тренировок, фильтрация, карточка тренировки, форма записи, экран подтверждения, мои бронирования, детали и отмена бронирования, уведомление об отмене, оценка инструктора.

### `4-design/`

* **data-model.md** — ER-модель и описание сущностей системы с указанием, какие данные приложение получает только для чтения из существующего бэкенда, а какие создаёт или изменяет самостоятельно.
* **api-sequence.md** — sequence-диаграммы основных сценариев взаимодействия с API (например, `createBooking` с вариантами успешного выполнения и обработки ошибок `201`, `409`, `410`).

### `5-mobile-app-spec/`

Финальный комплект технической документации для разработки клиентского мобильного приложения (роль пользователя — «Клиент»). Управление расписанием, тренировками и инструкторами находится вне границ проекта и доступно приложению только в режиме чтения.

* **00-INDEX.md** — индекс документации: перечень экранов, общей бизнес-логики и сквозных ограничений.
* **00-OPEN-QUESTIONS-LOG.md** — журнал открытых вопросов с текущими статусами.
* **SCR-1…SCR-9** — технические задания для каждого экрана, подготовленные по шаблону `_SCREEN_TEMPLATE.md`.
* **LOGIC-001…LOGIC-005** — описание общей бизнес-логики, используемой на нескольких экранах (статусы бронирования, правила отмены, push-уведомления, условия доступности записи и оценки), оформленное по шаблону `_LOGIC_TEMPLATE.md`.

### `api/`

* **openapi.yaml** — спецификация API клиентского мобильного приложения в формате OpenAPI 3.0.3: набор эндпоинтов, схемы данных, коды ответов и описание `securitySchemes`. Ресурсы расписания, тренировок и инструкторов предоставляются только для чтения, поскольку управляются существующей административной системой.

### `prompts/`

* **my-prompts.md** — полный перечень промптов, использованных для последовательной генерации всех материалов проекта — от анализа первоначального брифа до подготовки спецификации API.

## Как создавался проект

Все документы были подготовлены итеративно в ходе работы с бесплатными веб-версиями ChatGPT и Claude — без использования платных тарифов, API и сторонних аналитических инструментов. Каждый следующий документ создавался на основе предыдущих материалов, дополнялся уточняющими промптами и корректировался с учётом замечаний. Полная последовательность работы и использованные формулировки промптов приведены в файле [`prompts/my-prompts.md`](./prompts/my-prompts.md).
