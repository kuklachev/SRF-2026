# Аналитика проекта «Вертикаль» — приложение для записи на тренировки в скалодроме

Учебный/демонстрационный кейс по бизнес- и системному анализу: от сырого брифа
заказчика до готового пакета технических заданий и контракта API для
клиентского мобильного приложения скалодрома.

> **Важно.** Весь материал в этом репозитории получен с помощью бесплатных
> чат-версий **ChatGPT** и **Claude** — без платных подписок, без API-доступа
> и без специализированных no-code/аналитических инструментов. Все промпты,
> которые использовались для генерации файлов, зафиксированы в
> [`prompts/my-prompts.md`](./prompts/my-prompts.md).

## О чём проект

Владелица скалодрома «Вертикаль» ведёт запись клиентов на групповые
тренировки вручную (Telegram + тетрадка) и хочет мобильное приложение для
самостоятельной записи. Вся инфраструктура скалодрома (бэкенд, расписание,
админка) уже существует — предмет анализа и проектирования: **только
клиентское мобильное приложение и API для него**.

Материалы демонстрируют полный цикл аналитической работы:

1. выявление требований по сырому брифу заказчика;
2. формализация бизнес-, функциональных и нефункциональных требований;
3. постановки на дизайн экранов;
4. проектирование данных и API-взаимодействия;
5. финальный пакет ТЗ для разработки клиентского приложения;
6. спецификация API в формате OpenAPI.

## Структура репозитория

```
analysis/
├── 0-customer-brief/       # исходный (намеренно неполный) бриф заказчика
├── 1-elicitation/          # выявление требований: вопросы заказчику + описание домена
├── 2-requirements/         # формализованные требования
├── 3-design-brief/         # постановки на дизайн по каждому экрану
├── 4-design/               # модель данных и sequence-диаграммы API
├── 5-mobile-app-spec/      # итоговый пакет ТЗ для мобильного приложения
├── api/                    # контракт API (OpenAPI)
└── prompts/                # промпты, которыми получены все материалы
```

### `0-customer-brief/`
- **brief-climbing.md** — исходное письмо заказчика (Оли, владелицы
  скалодрома) с описанием проблемы. Намеренно неполное и местами
  противоречивое — отправная точка для анализа.

### `1-elicitation/`
- **customer-questions.md** — уточняющие вопросы заказчику и (по итогам
  обсуждения) его ответы.
- **domain-description.md** — цельное описание предметной области,
  собранное на основе брифа и ответов заказчика.

### `2-requirements/`
- **business-requirements.md** — бизнес-требования (BR) с ID, приоритетом и
  ссылкой на источник.
- **functional-requirements.md** — функциональные требования (FR): что
  система умеет делать.
- **non-functional-requirements.md** — нефункциональные требования (NFR):
  насколько хорошо система это делает (конкретные метрики, а не общие слова).
- **use-cases.md** — сценарии использования (UC).
- **user-stories.md** — пользовательские истории (US).

### `3-design-brief/`
- **00-screens-registry.md** — реестр всех экранов приложения.
- **01…09-*.md** — постановка на дизайн для каждого отдельного экрана
  (список тренировок, фильтры, карточка тренировки, форма записи,
  подтверждение, мои бронирования, детали/отмена, уведомление об отмене,
  оценка инструктора).

### `4-design/`
- **data-model.md** — ER-модель и описание сущностей системы, с пометкой,
  какие сущности приложение только читает (приходят из бэкенда), а какие
  создаёт/изменяет.
- **api-sequence.md** — sequence-диаграммы ключевых сценариев API
  (например, `createBooking` с ветками успеха и ошибок 201/409/410).

### `5-mobile-app-spec/`
Финальный пакет технических заданий на клиентское мобильное приложение
(роль «Клиент»; управление расписанием/тренировками/инструкторами — вне
скоупа, доступ только на чтение):
- **00-INDEX.md** — индекс пакета: перечень экранов, сквозной бизнес-логики
  и сквозных ограничений.
- **00-OPEN-QUESTIONS-LOG.md** — трекер открытых вопросов и их статус.
- **SCR-1…SCR-9** — ТЗ на каждый экран (по шаблону `_SCREEN_TEMPLATE.md`).
- **LOGIC-001…LOGIC-005** — сквозная бизнес-логика, используемая на
  нескольких экранах (статусы брони, политика отмены, push-уведомления,
  доступность записи и оценки — по шаблону `_LOGIC_TEMPLATE.md`).

### `api/`
- **openapi.yaml** — контракт API клиентского мобильного приложения
  (OpenAPI 3.0.3): эндпоинты, схемы, коды ответов, `securitySchemes`.
  Ресурсы расписания/тренировок/инструкторов доступны только на чтение,
  т.к. управляются существующей административной системой.

### `prompts/`
- **my-prompts.md** — полный список промптов, которыми последовательно
  генерировались все материалы выше (от анализа брифа до API-контракта).

## Как это делалось

Все документы созданы итеративно в диалоге с бесплатными веб-версиями
ChatGPT и Claude (без платных тарифов/подписок, без использования API):
каждый следующий файл строился на основе предыдущих через уточняющие
промпты и правки по замечаниям. Последовательность и формулировки
промптов — в [`prompts/my-prompts.md`](./prompts/my-prompts.md).
