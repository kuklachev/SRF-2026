# User Stories

> Этап 2. Пользовательские истории.
>
> Формат пользовательской истории:
>
> **«Как <роль>, я хочу <действие>, чтобы <ценность>.»**
>
> Каждая история сопровождается критериями приёмки и связана с функциональными требованиями.
>
> В рамках текущего проекта реализуется только роль **Клиент**.

# Общие критерии приёмки

Все пользовательские истории считаются реализованными при выполнении следующих условий:

- функциональность соответствует связанным функциональным требованиям;
- пользователь может выполнить сценарий без обращения к администратору;
- отображаемые данные получаются через существующий backend;
- действия пользователя завершаются понятным сообщением об успешном выполнении либо возникшей ошибке.

---

# Вне текущего скоупа

В рамках текущей версии приложения пользовательские истории **не разрабатываются** для следующих ролей:

- Администратор;
- Инструктор.

Также не входят в текущую поставку:

- управление расписанием;
- создание тренировок;
- изменение тренировок;
- управление инструкторами;
- онлайн-оплата;
- программа лояльности.
