# Функциональные требования

> Этап 2. Что система должна делать (функции и поведение).
> Документ составлен на основании описания домена (domain-description.md).

Приоритеты требований указаны по модели **MoSCoW**:

- **Must** — обязательно для реализации в MVP;
- **Should** — желательно реализовать при наличии времени;
- **Could** — возможно реализовать в дальнейшем;
- **Won't** — не входит в текущую поставку.

---

# Границы скоупа

В рамках проекта разрабатывается **только клиентское мобильное приложение**.

Все операции по управлению тренировками, расписанием, инструкторами и прокатным оборудованием выполняются существующей административной системой и доступны приложению только через API.

Следствия:

- расписание создаётся существующей системой;
- клиентское приложение не изменяет тренировки;
- инструкторы отображаются только как справочная информация;
- контроль свободных мест выполняется сервером.

# Ограничения функциональности

Следующие функции находятся **вне текущего скоупа проекта** и реализуются существующей инфраструктурой:

- создание тренировок;
- изменение расписания;
- управление инструкторами;
- управление стоимостью тренировок;
- контроль прокатного оборудования;
- онлайн-оплата;
- административная панель;
- интерфейс инструктора.

Клиентское приложение получает необходимые данные исключительно через API существующего backend.
