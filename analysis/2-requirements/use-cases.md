## UC-1: Просмотр расписания тренировок

**Приоритет:** Must  
**Тип:** Основной

### Описание
Клиент просматривает список доступных тренировок.

### Основной актор
Клиент

### Предусловия
- Приложение запущено
- Доступен backend

### Основной сценарий
1. Клиент открывает экран расписания.
2. Система запрашивает данные у backend.
3. Система отображает список тренировок с ключевой информацией (дата, время, инструктор, стоимость, места).
4. Клиент получает актуальное расписание.

### Постусловия
- Данные расписания отображены пользователю

### Связанные требования
- BR-4
- FR-1, FR-2, FR-4
- US-1

### Нефункциональные ограничения
- NFR-1 (≤ 2 секунды загрузка списка)
- NFR-6 (только актуальные данные)

## UC-4: Запись на тренировку

**Приоритет:** Must  
**Тип:** Ключевой бизнес-процесс

### Описание
Клиент создаёт бронирование на тренировку.

### Основной актор
Клиент

### Основной сценарий
1. Клиент выбирает тренировку.
2. Система проверяет наличие свободных мест (backend).
3. Клиент выбирает тип оборудования (личное или прокатное).
4. Клиент указывает количество участников (до 3 человек).
5. Клиент подтверждает запись.
6. Система создаёт бронь.

### Альтернативные сценарии
- Нет свободных мест → создание брони невозможно

### Постусловия
- Бронь создана и отображается в списке пользователя

### Связанные требования
- BR-2, BR-6, BR-7, BR-11
- FR-5, FR-6, FR-7, FR-8, FR-9, FR-10
- US-4, US-5, US-6

### Нефункциональные ограничения
- NFR-2 (≤ 3 секунды выполнение операции)
- NFR-4 (исключение двойных бронирований)

## UC-6: Отмена бронирования

**Приоритет:** Must  
**Тип:** Основной

### Описание
Клиент отменяет ранее созданную бронь.

### Основной сценарий
1. Клиент выбирает бронирование.
2. Система проверяет время до начала тренировки.
3. Клиент подтверждает отмену.
4. Система отменяет бронь.

### Альтернативные сценарии
- Менее 2 часов до тренировки → фиксируется поздняя отмена

### Постусловия
- Место освобождается (если применимо)
- Статус брони обновляется

### Связанные требования
- BR-5
- FR-13, FR-14, FR-15, FR-16
- US-8, US-9

### Нефункциональные ограничения
- NFR-5 (моментальное освобождение мест)

## UC-7: Уведомление об отмене тренировки

**Приоритет:** Must  
**Тип:** Системный (инициируется backend)

### Основной актор
Backend / Система скалодрома

### Описание
Система уведомляет клиента об отмене тренировки.

### Основной сценарий
1. Backend фиксирует отмену тренировки.
2. Система изменяет статус брони.
3. Отправляется push-уведомление клиенту.
4. Клиент получает уведомление.

### Постусловия
- Бронь имеет статус «Отменена скалодромом»

### Связанные требования
- BR-8
- FR-17, FR-18, FR-19
- US-10, US-11

---

## UC-8: Оценка инструктора

**Приоритет:** Should  
**Тип:** Дополнительный

### Описание
Клиент оставляет оценку и отзыв после тренировки.

### Основной сценарий
1. Тренировка завершена.
2. Система открывает возможность оценки.
3. Клиент ставит оценку (1–5) и пишет отзыв.
4. Система сохраняет данные.

### Альтернативные сценарии
- Тренировка не завершена → оценка недоступна

### Связанные требования
- BR-10
- FR-20, FR-21, FR-22
- US-12

# 3. Общие ограничения (применяются ко всем UC)

## Технические ограничения
- Все данные получаются только через backend API
- Клиент не изменяет расписание и тренировки (NFR-14)
- Backend является единственным источником данных (BR-11, NFR-6)

## Безопасность
- Доступ только к собственным данным пользователя (NFR-9)
- Передача данных только по HTTPS (NFR-8)

## Производительность
- Загрузка расписания ≤ 2 секунд (NFR-1)
- Операции бронирования ≤ 3 секунд (NFR-2)
