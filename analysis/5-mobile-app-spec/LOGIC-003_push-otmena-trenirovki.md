## Обзор

Когда скалодром отменяет тренировку (административная система, вне скоупа
этого приложения — NFR-14), клиенты с активной бронью на неё автоматически
получают push-уведомление (FR-17, BR-8), а статус их брони меняется на
«Отменена скалодромом» (FR-18). Приложение должно корректно провести
пользователя из уведомления к деталям соответствующей брони и гарантировать,
что клиент увидит актуальный статус, даже если push не был доставлен или не
был открыт.

### User Story

> Как клиент, я хочу сразу узнать, если скалодром отменил тренировку, на
> которую я записан, и понимать статус своей брони без дополнительных
> действий.

### Бизнес-ценность

- Своевременное информирование снижает количество клиентов, которые придут
  на отменённую тренировку впустую.
- Единый источник истины (backend) исключает ситуацию, когда статус брони в
  приложении расходится с реальностью, если push не дошёл (NFR-6).

## Описание логики

### Шаг 1: Получение события backend

Источник события — административная система скалодрома; клиентское
приложение не инициирует и не может отменить тренировку (NFR-14, только
чтение). Push отправляется автоматически всем клиентам, имеющим бронь со
статусом `active` на отменяемую тренировку.

### Шаг 2: Отображение в зависимости от состояния приложения

- Приложение в фоне/закрыто → стандартное системное push-уведомление ОС с
  кратким текстом (дата и время тренировки).
- Приложение открыто (foreground) → желательно показать in-app плашку
  (баннер) поверх текущего экрана; конкретный визуальный механизм — на
  усмотрение дизайна (открытый вопрос из 08-cancellation-notification.md).

### Шаг 3: Маршрутизация по тапу

Тап по уведомлению открывает:
- SCR-7 (детали конкретной брони), если `booking_id` присутствует в пейлоаде;
- SCR-6 (список бронирований) — fallback, если `booking_id` не удалось
  определить.

### Шаг 4: Fallback без взаимодействия с push

Независимо от того, было ли уведомление показано/открыто/пропущено, клиент
при обычном заходе в SCR-6 или SCR-7 всегда увидит актуальный статус
«Отменена скалодромом», так как источник истины — ответ backend
(`GET /bookings`, `GET /bookings/{bookingId}`), а не локальное состояние
пуш-уведомлений (NFR-6).

### Шаг 5: Блокировка повторных действий

На SCR-1/SCR-3 отменённая скалодромом тренировка отображается как
недоступная для записи (`training.status = cancelled`, FR-19). На SCR-6/SCR-7
соответствующая бронь отображается только для просмотра — кнопка «Отменить
бронирование» скрывается ([LOGIC-002](./LOGIC-002_politika-otmeny.md) не
применяется к терминальному статусу `club_cancelled`).

## Критерии приёмки

| ID | Критерий |
|----|----------|
| AC-001 | **Дано** тренировка отменена скалодромом и у клиента активная бронь, **Когда** событие происходит, **Тогда** клиент получает push-уведомление |
| AC-002 | **Дано** push содержит `booking_id`, **Когда** клиент тапает по уведомлению, **Тогда** открывается SCR-7 с деталями соответствующей брони |
| AC-003 | **Дано** push не содержит `booking_id`, **Когда** клиент тапает по уведомлению, **Тогда** открывается SCR-6 |
| AC-004 | **Дано** push не был доставлен или не открыт, **Когда** клиент самостоятельно заходит на SCR-6/SCR-7, **Тогда** статус «Отменена скалодромом» всё равно отображается корректно |
| AC-005 | **Дано** бронь имеет статус `club_cancelled`, **Когда** отображаются SCR-6/SCR-7, **Тогда** действие «Отменить бронирование» недоступно |
| AC-006 | **Дано** тренировка отменена, **Когда** клиент открывает её карточку на SCR-1/SCR-3, **Тогда** запись недоступна и отображается статус «Тренировка отменена» |

## Обработка ошибок

| Тип ошибки | Контекст | Действие |
|------------|----------|----------|
| Push не доставлен (сбой пуш-провайдера, устройство offline) | Вне контроля клиентского приложения | Полагаться на fallback через SCR-6/SCR-7 (Шаг 4); журнал уведомлений не предусмотрен в MVP |
| `booking_id` в пейлоаде указывает на несуществующую/чужую бронь | Ошибка на стороне backend/push-сервиса | При 403/404 от `GET /bookings/{bookingId}` — направить на SCR-6 вместо ошибки |
