## Обзор

Экран результата успешного оформления брони. Данные для отображения приходят
целиком в ответе `POST /bookings` с SCR-4 — отдельного запроса на этом
экране не требуется. Экран не предполагает состояния ошибки: вся обработка
ошибок выполняется на SCR-4 до перехода сюда.

### User Story

> Как клиент, я хочу сразу увидеть подтверждение того, что моя запись
> оформлена, с кратким резюме брони.

### Бизнес-ценность

- Формирует доверие к процессу записи явным подтверждением результата.
- Точка возможного информирования о политике отмены — потенциальное
  улучшение показателя доли самостоятельных отмен (M-3), требует
  согласования с продуктом.

## Макет экрана

### Структура

```
┌─────────────────────────────────────┐
│           ✓ Запись оформлена        │  ← Иконка успеха + заголовок
├─────────────────────────────────────┤
│ Статус: Подтверждена                │
│ Тренировка: дата, время             │
│ Участников: N                       │
│ Оборудование: своё / прокат         │
```

| Компонент | Описание | Обязательность |

## Критерии приёмки

### Позитивные сценарии

| ID | Критерий | Приоритет |
|----|----------|-----------|
| AC-001 | **Дано** бронь успешно создана, **Когда** открывается SCR-5, **Тогда** отображается статус «Подтверждена» и корректная сводка брони | P0 |
| AC-002 | **Дано** клиент на SCR-5, **Когда** он нажимает «Мои бронирования», **Тогда** открывается SCR-6 и созданная бронь присутствует в списке | P0 |

### Граничные условия

| ID | Критерий | Приоритет |
|----|----------|-----------|
| AC-E01 | **Дано** клиент закрыл приложение сразу после SCR-5, **Когда** он открывает приложение снова, **Тогда** статус брони на SCR-6 актуален независимо от того, был ли просмотрен SCR-5 | P2 |
