## Входные данные

| Название | Тип | Возможные значения | Описание |
|----------|-----|-------------------|----------|
| `booking.status` | Состояние (из ответа API) | `active`, `cancelled`, `late_cancel`, `club_cancelled` | Статус брони, приходит в объекте `Booking` (см. `openapi.yaml` → `BookingStatus`) |
| `booking.payment_status` | Состояние | `unpaid`, `paid` | Статус офлайн-оплаты, отображается отдельным элементом рядом со статусом брони |
| `training.start_at` | Состояние | ISO date-time | Используется, чтобы отличить предстоящую бронь от прошедшей (для группировки на SCR-6) |

## Обзор

Единая логика сопоставления серверного значения `booking.status` (и производного
признака "предстоящая/прошедшая") с текстом, цветом и иконкой бейджа статуса,
который отображается на SCR-6 (список) и SCR-7 (детали). Централизация нужна,
чтобы формулировки статусов не расходились между экранами.

### User Story

> Как клиент, я хочу видеть понятный и одинаковый по всему приложению статус
> моей брони, чтобы точно понимать, действует ли запись, отменена ли она мной,
> поздно, или скалодромом.

### Бизнес-ценность

- Единообразие статусов снижает количество обращений в поддержку по вопросу
  «что означает этот статус».
- Явное отображение статуса «Отменена скалодромом» закрывает FR-18.
- Согласованность между SCR-6/SCR-7/SCR-8 обязательна, так как статус может
  быть впервые увиден пользователем на любом из трёх экранов (push, список,
  детали) — NFR-6 (backend всегда источник истины).

## Описание логики

### Шаг 1: Определение базового статуса

Берётся значение `booking.status` из ответа `GET /bookings` или
`GET /bookings/{bookingId}`. Ветвление по четырём серверным значениям — иных
значений API не возвращает (см. `BookingStatus` в `openapi.yaml`).

### Шаг 2: Уточнение для статуса `active`

Поскольку сервер не возвращает отдельного статуса «Завершена» (в `openapi.yaml`
это не отдельный enum-статус, а производное состояние), клиент обязан
самостоятельно сравнить `training.start_at + training.duration_min` с текущим
временем устройства **только для целей визуальной группировки/бейджа**, не для
бизнес-решений (проверки типа «можно ли отменить» всегда выполняются сервером).

- Если тренировка ещё не наступила → бейдж «Подтверждена» (зелёный).
- Если тренировка уже прошла, а статус всё ещё `active` → бейдж «Завершена»
  (нейтральный синий/серый), доступен CTA «Оценить инструктора», если
  `has_rating = false` (см. [LOGIC-005](./LOGIC-005_dostupnost-ocenki.md)).

### Шаг 3: Отображение статуса оплаты

Статус оплаты (`payment_status`) отображается отдельным, менее заметным
элементом рядом с основным статусом брони, не заменяет и не смешивается с ним:
`unpaid` → «Оплата на месте», `paid` → «Оплачено». Клиент не может изменить это
значение (BR-13, только офлайн-оплата).

## Критерии приёмки

| ID | Критерий |
|----|----------|
| AC-001 | **Дано** `booking.status = active` и тренировка в будущем, **Когда** отображается бейдж, **Тогда** показывается «Подтверждена» (зелёный) |
| AC-002 | **Дано** `booking.status = active` и тренировка уже прошла, **Когда** отображается бейдж, **Тогда** показывается «Завершена» |
| AC-003 | **Дано** `booking.status = cancelled`, **Когда** отображается бейдж, **Тогда** показывается «Отменена клиентом» |
| AC-004 | **Дано** `booking.status = late_cancel`, **Когда** отображается бейдж, **Тогда** показывается «Поздняя отмена» без указания штрафа (FR-16 — штраф не начисляется) |
| AC-005 | **Дано** `booking.status = club_cancelled`, **Когда** отображается бейдж на SCR-6/SCR-7, **Тогда** показывается «Отменена скалодромом» (красный), действия отмены скрыты |
| AC-006 | **Дано** любое значение статуса, **Когда** бейдж отображается на SCR-6 и на SCR-7 для одной и той же брони, **Тогда** текст и цвет идентичны |

## Обработка ошибок

| Тип ошибки | Контекст | Действие |
|------------|----------|----------|
| Неизвестное значение `status` (не входит в enum) | Ответ API содержит значение вне `BookingStatus` | Отобразить нейтральный бейдж «Статус уточняется» и залогировать инцидент для аналитики; не блокировать остальной UI |
