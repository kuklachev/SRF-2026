## Сквозные экраны и состояния (применяются ко всем SCR)

Для каждого экрана, взаимодействующего с backend (все экраны, кроме элементов
локального UI), должны быть предусмотрены состояния:

- **Загрузка** — индикатор ожидания ответа backend (см. NFR-1, NFR-2).
- **Пустое состояние** — данные отсутствуют (например, нет доступных
  тренировок, нет бронирований).
- **Ошибка** — недоступен backend или превышено время ожидания; сообщение
  должно быть понятным пользователю (см. «Общие критерии приёмки» в
  user-stories.md).
- **Успех** — понятное подтверждающее сообщение о результате действия.

Общие ограничения дизайна:

- Интерфейс адаптирован под смартфоны, без горизонтальной прокрутки (NFR-11).
- Клиент видит только собственные данные (NFR-9).
- Клиентское приложение не предоставляет средств редактирования тренировок,
  расписания и инструкторов (NFR-14) — только просмотр справочной информации.
- Онлайн-оплата не реализуется (BR-13) — статус оплаты в интерфейсе только
  отображается, способ оплаты не выбирается.

## Экраны вне текущей проработки (зафиксировать как открытый вопрос)

В business-requirements.md «Границы проекта» регистрация клиента указана как
часть скоупа, однако ни functional-requirements.md, ни use-cases.md, ни
user-stories.md не описывают функциональные требования или сценарий
регистрации/авторизации клиента (use-cases.md прямо указывает: «регистрация
пользователя — нет FR-описания»).

**Рекомендация:** уточнить у заказчика/аналитика необходимость экрана
регистрации/входа (SCR-0) до начала дизайна, либо явно зафиксировать, что
доступ клиента в приложение обеспечивается механизмом, не входящим в текущий
скоуп проектирования (например, уже существующим у скалодрома способом
идентификации). До получения ответа экран не включён в постановки на дизайн.
