# SCR-2. Фильтры расписания

## Назначение экрана
Позволяет клиенту задать параметры фильтрации, чтобы быстрее найти
подходящую тренировку в списке (SCR-1).

## Роль пользователя
Клиент.

## Тип экрана
Модальный экран / выдвижная панель (bottom sheet) поверх SCR-1 — итоговое
архитектурное решение (модалка vs отдельный экран) принимает дизайнер, в
требованиях явно не зафиксировано.

## Предусловия
- Клиент находится на экране SCR-1.

## Точка входа
- Иконка/кнопка «Фильтры» на SCR-1.

## Состав информации на экране
Требования (FR-4) не детализируют конкретный набор фильтров, поэтому набор
параметров формируется исходя из доступных полей тренировки (FR-2):
- дата / диапазон дат;
- время суток (например, утро/день/вечер) или диапазон времени;
- формат тренировки (в т.ч. «для новичков»);
- инструктор;
- наличие свободных мест («показывать только тренировки со свободными
  местами»).

## Действия пользователя
- Выбор одного или нескольких параметров фильтрации.
- Кнопка «Применить» — возвращает на SCR-1 с обновлённым списком.
- Кнопка «Сбросить» — очищает все выбранные фильтры.
- Закрытие экрана без применения изменений (крестик/свайп) — фильтры не
  сохраняются.

## Состояния экрана
- **По умолчанию**: фильтры не заданы либо отображены ранее применённые
  значения (если пользователь возвращается для редактирования).
- **Заполнено частично/полностью**: визуально выделены выбранные значения.
- Применение фильтра инициирует загрузку списка на SCR-1 — тайминг
  соответствует требованию к обновлению списка (см. NFR-1 для экрана
  расписания).

## Ограничения и особые случаи
- Фильтрация выполняется на основании данных, полученных от backend —
  экран фильтров не хранит и не создаёт собственные тренировки (NFR-14).
- Экран должен быть простым для использования на одном экране без
  горизонтальной прокрутки (NFR-11).

## Связанные требования
- Функциональные требования: FR-4.
- Пользовательские истории: US-3.
- Use Case: UC-3.

## Открытые вопросы для дизайна
- Точный перечень и приоритет фильтров не зафиксирован в требованиях —
  рекомендуется согласовать с заказчиком минимальный набор для MVP (риск
  избыточной сложности при ограниченном сроке разработки — BR-12).
- Нужно ли сохранять последние применённые фильтры между сессиями? Не
  описано, для MVP предполагается сброс при перезапуске приложения.
